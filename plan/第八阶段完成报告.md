## 四、LLM 输入包含哪些上下文

LLM 收到的 system + user prompt 包含：

1. **角色定义**：「你是一个小说创作系统的意图理解模块」
2. **召回的相关文件**：`retrieveContext` 返回的 path + excerpt（每条截断到 200 字符）
3. **创作指南摘要**：`readStyleGuideSummary` 返回的内容（截断到 500 字符）
4. **支持的动作类型说明**：`gender_change` 和 `relationship_change` 的字段定义
5. **用户原始输入**
6. **强约束**：「只输出 JSON，不要输出任何解释或 markdown 格式」

## 五、LLM JSON 输出结构

```json
{
  "understood": ["将林晓性别改为女", "将林晓和陈磊关系改为敌对"],
  "missing": ["第七章中林晓是否也需要同步修改？"],
  "actions": [
    { "type": "gender_change", "character": "林晓", "target": "女" },
    { "type": "relationship_change", "charA": "林晓", "charB": "陈磊", "relationship": "敌对" }
  ]
}
```

### zod 校验

- `understood`: `z.array(z.string())`
- `missing`: `z.array(z.string())`
- `actions`: `z.array(z.union([gender_action, relationship_action]))`
- `gender_action`: `{ type: "gender_change", character: string, target: "男"|"女" }`
- `relationship_action`: `{ type: "relationship_change", charA: string, charB: string, relationship: 7 种关系之一 }`

## 六、fallback 策略

```
messages API 收到用户消息
  ↓
isLlmEnabled() ?
  ├─ YES → generateActionPlanWithLlm()
  │        ├─ 成功 + actions 非空 → 使用 LLM plan
  │        ├─ 成功 + actions 为空 → 使用 LLM understood/missing，无 plan
  │        ├─ zod 校验失败 → 回退规则解析
  │        ├─ JSON 解析失败 → 回退规则解析
  │        └─ API 调用失败 → 回退规则解析
  └─ NO → 直接规则解析
  ↓
规则解析（正则匹配 gender/relationship）
  ↓
组装 message + plan 返回
```

**关键保证**：LLM 失败永远不会导致 messages API 报错，始终回退到规则解析。

## 七、messages API 如何变化

### 新增步骤

```
1. retrieveContext(bookId, content)         ← 不变
2. readStyleGuideSummary(bookId)            ← 不变
3. isLlmEnabled() ?                         ← 新增
   ├─ YES: generateActionPlanWithLlm(input)
   │   ├─ 成功: 使用 LLM actions 生成 plan
   │   └─ 失败: 回退规则
   └─ NO: 规则解析
4. 组装 message                              ← 变化
```

### message.brief 变化

| 字段 | 来源 |
|------|------|
| `understood` | LLM 输出的 `understood` 数组，或规则解析生成的摘要 |
| `contextPaths` | `retrieveContext` 返回的 paths + `skills/style_guide_summary.md` |
| `missing` | LLM 输出的 `missing` 数组（规则解析时为空） |

### 响应示例（LLM 模式）

```json
{
  "message": {
    "id": "m-xxx",
    "role": "assistant",
    "content": "已根据最近改动和相关设定拟好变更计划,请在右侧确认。",
    "createdAt": "14:32",
    "brief": {
      "understood": ["将林晓性别改为女"],
      "contextPaths": ["人物设定/林晓.md", "skills/style_guide_summary.md"],
      "missing": []
    }
  },
  "plan": [{ "...ActionNode 树..." }]
}
```

## 八、没有 API key 时是否仍可运行

**是。** 完全兼容：

- `isLlmEnabled()` 检查 `MIMO_API_KEY` 或 `DEEPSEEK_API_KEY` 是否存在
- 如果 key 缺失，`getConfig()` 返回 `null`，`generateActionPlanWithLlm` 返回 `null`
- messages API 检测到 `null` 后直接走规则解析
- 所有现有功能（mock 回退、规则解析、工作台、ActionPlan 确认/放弃）不受影响

## 九、如何手动验证

### 1. 无 API key 时（验证回退）

```bash
# 不设置任何 API key，直接启动
pnpm dev

# 发送消息，应走规则解析
curl -X POST http://localhost:3000/api/books/demo-guixu/messages \
  -H "Content-Type: application/json" \
  -d '{"content":"把林晓改成女的"}'

# 应返回 plan（规则解析结果）
```

### 2. 有 API key 时（验证 LLM）

```bash
# 创建 .env 并填入 key
echo "LLM_PROVIDER=mimo" > .env
echo "MIMO_API_KEY=your-key-here" >> .env

pnpm dev

# 发送自然语言消息（规则解析无法处理的）
curl -X POST http://localhost:3000/api/books/demo-guixu/messages \
  -H "Content-Type: application/json" \
  -d '{"content":"林晓的性别改成女性，另外她跟陈磊应该是敌对关系"}'

# 应返回 LLM 解析结果，brief.understood 包含 LLM 理解的任务
```

### 3. 验证 DeepSeek provider

```bash
# .env
LLM_PROVIDER=deepseek
DEEPSEEK_API_KEY=your-key-here

# 启动后发送消息
pnpm dev
```

### 4. 验证 zod 校验失败回退

```bash
# 设置一个错误的 base URL 强制 LLM 返回异常
MIMO_BASE_URL=https://httpbin.org/post
MIMO_API_KEY=test

# 发送消息，应静默回退到规则解析
curl -X POST http://localhost:3000/api/books/demo-guixu/messages \
  -H "Content-Type: application/json" \
  -d '{"content":"把林晓改成女的"}'
```
