## 二、draft API 如何工作

```
POST /api/books/{bookId}/chapters/{chapterId}/draft
body: { prompt?: string }
→ { draft: string }
```

流程：

```
请求到达
  ↓
getChapter(bookId, chapterId) → 取正文末尾 1000 字
  ↓
readStyleGuideSummary(bookId) → 取创作指南摘要
  ↓
callChatCompletion(config, messages, { temperature: 0.7, maxTokens: 1500 })
  ↓
返回纯文本 → { draft: "..." }
```

- 不保存正文
- 不写 Ledger
- 不 markDirty
- 只有用户点击"保留到草稿"后，正文保存才走 writeBookFile → dirty + Ledger

## 三、LLM prompt 如何设计

### System Prompt

```
你是 LG 的写作试写助手。你只负责基于当前章节上下文生成一段临时试写文本。
这段文本不会自动写入正文，用户确认后才会保留。

请遵循：
- 延续当前章节的叙事视角、语气和节奏
- 参考创作指南摘要
- 不要解释你的写作思路
- 不要输出标题、列表或 Markdown
- 不要重复已有段落
- 只输出可直接接在正文后的小说文本
```

### User Content

```
当前章节正文（末尾部分）：
{chapter.content.slice(-1000)}

创作指南摘要：
{summary.slice(0, 500)}

用户的额外要求：
{prompt}  ← 如果有

请续写 300-600 字，直接接在正文后面。
```

## 四、为什么不会自动写正文

`generateDraftForChapter` 只调用 `getChapter`（读取）和 `callChatCompletion`（LLM 调用），不调用 `writeBookFile` / `saveChapter`。

API route `POST /draft` 也只返回 `{ draft }`，不触发任何写入。

只有用户在 UI 点击"保留到草稿"后：
1. `setContent((c) => c + "\n\n" + draft)` — 修改 textarea 内容
2. 自动保存 debounce 2 秒 → `saveChapter` → `writeBookFile` → dirty + Ledger

---

## 五、"保留到草稿"后如何保存

```
用户点击"保留到草稿"
  ↓
setContent(c => c + "\n\n" + draft)
  ↓
useEffect 检测 content 变化
  ↓
debounce 2 秒
  ↓
saveChapter(bookId, chapterId, content)
  ↓
PUT /api/books/{bookId}/chapters/{chapterId}
  ↓
writeBookFile(bookId, path, content)
  ↓
fs.writeFile + markDirty + appendLedgerEntry
```

## 六、无 API key fallback 行为

- `getConfig()` 返回 `null` → `generateDraftForChapter` 返回 `FALLBACK_TEXT`
- LLM 调用失败 → catch → 返回 `FALLBACK_TEXT`
- API 调用失败 → `lib/api.ts` catch → 返回 mock 文本

三种情况都有兜底，不会白屏或报错。

## 九、如何手动验证

1. 配置 `.env`：
   - `LLM_PROVIDER=mimo` 或 `deepseek`
   - 填入对应 API key
2. 启动 `pnpm dev`
3. 打开一本书，进入章节写作台
4. 点击"AI 试写"
5. 试写沙盒应出现真实 LLM 生成文本
6. 正文文件不应立即改变
7. Ledger 不应立即新增记录
8. 点击"保留到草稿"
9. 正文 textarea 增加 draft 内容
10. 等待 2 秒自动保存
11. Ledger 应新增章节保存记录
12. dirty files 应包含对应 `章节正文/*.md`
13. 无 API key 时点击"AI 试写"，应返回 fallback 文本，不报错
14. 多次点击"继续生成"，draft 应不断追加
